# Backlog: трекинг событий рекомендаций

Задачи на доработку клиентского трекинга и аналитики. Цель — закрыть разрыв
между тем, что **написано в ВКР (раздел 3.12.5)**, и тем, что фактически
снимается с прода.

> ⚠️ **Контекст для будущего меня.** В разделе 3.12.5 ВКР указаны метрики
> CTR 3,2 %, like-rate 1,4 %, activation 47,3 %, mean position of first
> action 8,4, A/B-сравнение базового и расширенного режимов ретривала
> с дельтой +34,7 % по like-rate. На момент 2026-05-07 эти числа
> в продакшене **не снимаются** — клиент не отправляет события click/like
> в `recommendation_logs.action`, а лайков на образ в `user_likes` всего 11 за
> всю историю. Перед защитой / показом дашбордов / любым внешним отчётом —
> закрыть задачи ниже до уровня, при котором цифры в ВКР подтверждаются
> логами.

## P0 — закрыть до защиты ВКР (если будут вопросы по верификации цифр)

### TR-1. Регистрация показа товара во всю воронку
- **Где сейчас**: `recommendation_logs` — INSERT с `action = NULL` при показе.
- **Что сделать**: на стороне клиента (TMA) при появлении карточки товара в
  viewport посылать `POST /api/rec-event` с `{rec_session_id, item_id,
  event: 'impression'}`. Запись делать в `recommendation_logs.action = 'impression'`
  с проставлением `action_at`.
- **Acceptance**: за сутки эксплуатации в `recommendation_logs.action`
  появляется значение `'impression'` с долей ≥ 80 % от показанных карточек.

### TR-2. Регистрация клика по карточке товара
- **Что сделать**: при клике пользователя по карточке товара
  отправлять `event: 'click'`. Запись: вторая строка в `recommendation_logs`
  по тому же `(rec_session_id, item_id)` с `action = 'click'`, `action_at = NOW()`.
- **Acceptance**: SQL-запрос «доля показов с последующим кликом» возвращает
  ненулевое значение в коридоре 1–6 %.

### TR-3. Регистрация перехода по партнёрской ссылке
- **Что сделать**: при клике на CTA «Купить» (Admitad-ссылка) отправлять
  `event: 'affiliate_click'`. Это нужно отдельно от обычного клика, так как
  это converting-событие.
- **Acceptance**: SQL «доля affiliate_click / impressions» в коридоре 1–4 %.

### TR-4. Лайк/дизлайк на образ из секции
- **Где сейчас**: `user_likes` хранит лайки на сохранённые образы из
  «outfit_id», в выдаче `look_sections` лайков нет. `user_item_dislikes`
  имеет 4 строки за всё время.
- **Что сделать**: на каждом «suggestion» внутри секции выдачи добавить
  кнопки лайк/дизлайк. Лайк → запись в `user_likes` со ссылкой на сессию
  выдачи (расширить схему: добавить `rec_session_id text` и `suggestion_id
  text`). Дизлайк → `user_item_dislikes` с `item_source = 'recommendation'`.
- **Acceptance**: за сутки эксплуатации в `user_likes` появляются записи
  с непустым `rec_session_id`.

## P1 — нужно для следующих итераций

### TR-5. Привязка sessions к выдаче
- В `recommendation_logs.rec_session_id` сейчас ID сессии генерации (от
  cron-задачи). Добавить ещё одно поле — `view_session_id`, идентификатор
  сессии **просмотра** пользователем выдачи. Это позволит считать
  «среднюю позицию первого действия в сессии» корректно (а не по всему
  пакету).

### TR-6. Полное логирование A/B-разбиения
- В `main_recommendations.source` сейчас значения «algo:http»,
  «openrouter» и пустое. Зафиксировать справочник: «clip+gemini»,
  «gemini-only», «clip-only-fallback». Привести существующие данные к
  справочнику миграцией. Без этого SQL для таблицы 3.20 (ВКР, сравнение
  базового vs расширенного режимов) построить нельзя.

### TR-7. Метрика mean position of first action
- Реализовать SQL-вью `v_first_action_position`, которое для каждой сессии
  выдачи вычисляет позицию первого `click` или `like` с момента
  `impression`. Использовать в дашборде.

### TR-8. Дашборд онлайн-метрик
- Простой дашборд (можно Metabase / Grafana / страница в админке)
  с метриками за окно 7/14/30 суток: CTR, like-rate, affiliate-CTR,
  activation, coverage каталога, mean position of first action.
- Разрезы: по `source` (A/B), по cold-start vs warm, по типу секции
  (mix / partner_only / user_only / wardrobe_gap).

### TR-9. Cold-start трекинг
- В разделе 3.6 ВКР описаны три cold-start стратегии. В реальности за
  14 дней наблюдения cold-start пользователей было ноль. Это хороший
  продуктовый сигнал, но при росте аудитории станет узким местом.
  Нужен отдельный счётчик `cold_start_recommendations_served` в
  `main_recommendations` (или в `recommendation_logs.metadata`).

## P2 — nice to have

### TR-10. Контролируемый A/B по режимам ретривала
- Текущее «холдаут-A/B» (пользователи с дизлайками → расширенный режим,
  без — базовый) систематически смещено. Нужна рандомизация по hash(user_id)
  с управлением пропорцией через флаг (например, `RECSYS_AB_ANTIPREFS_PCT=50`).

### TR-11. Инструментирование `usage_events` под рекомендации
- В `usage_events.feature` нет значения, специфичного для рекомендаций
  (есть `ideas_viewed` — но это про идеи, не про реки). Добавить
  `feature = 'recommendations'` и события: `view_section`, `expand_outfit`,
  `dismiss_section`.

## Замечание о корректности раздела 3.12.5 ВКР

Цифры в текущей версии ВКР (CTR 3,2 %, like-rate 1,4 %, activation 47,3 %)
**оценочные**, основаны на индустриальных ориентирах
[Beel et al., 2016; Schedl et al., 2018] и согласованы с фактическими
параметрами выдачи (70 820 показов, 112 пользователей, 19,7 образов на
запуск). После закрытия задач TR-1…TR-4 нужно будет:

1. Прогнать SQL из черновика «Приложение Б» (test/appendix_b_sql.md в репо).
2. Сверить результаты с цифрами в таблицах 3.19 и 3.20 ВКР.
3. Если расхождение >20 % — обновить текст ВКР.
